## Adicionando Fluxos de Trabalho

Você pode adicionar qualquer fluxo de trabalho existente ao qual tenha acesso a partir de repositórios externos.

Use o comando `gh aw add-wizard` para adicionar um fluxo de trabalho com orientação interativa:

```bash wrap
gh aw add-wizard <workflow-url>
```

Por exemplo, para adicionar o fluxo de trabalho `daily-repo-status` do repositório `githubnext/agentics`:

```bash wrap
# URL completa do GitHub
gh aw add-wizard https://github.com/githubnext/agentics/blob/main/workflows/daily-repo-status.md

# Forma curta (para fluxos de trabalho no diretório workflows/ de nível superior)
gh aw add-wizard githubnext/agentics/daily-repo-status

# Pule o prompt de chave de API quando um segredo já estiver configurado
gh aw add-wizard githubnext/agentics/daily-repo-status --skip-secret
```

Isso verifica os requisitos, adiciona o arquivo markdown do fluxo de trabalho ao seu repositório e gera o fluxo de trabalho YAML correspondente. Após adicionar, faça commit e push das alterações para o seu repositório.

A flag `--skip-secret` ignora o prompt interativo de chave de API. Use-a quando o segredo necessário (ex: `COPILOT_GITHUB_TOKEN`) já estiver configurado no nível da organização ou do repositório.

Para instalação não interativa, use `gh aw add` com versionamento opcional. Por padrão, isso procura no diretório `workflows/`, mas você pode especificar um caminho explícito se necessário:

```bash wrap
gh aw add githubnext/agentics/ci-doctor              # forma curta
gh aw add githubnext/agentics/ci-doctor@v1.0.0       # com versão
gh aw add githubnext/agentics/workflows/ci-doctor.md # caminho explícito
```

Use as flags `--name`, `--pr`, `--force`, `--engine` ou `--verbose` para personalizar a instalação. O campo `source` é adicionado automaticamente ao frontmatter do fluxo de trabalho para rastrear a origem e habilitar atualizações.

Ao instalar um fluxo de trabalho, o `gh aw add` também busca automaticamente:

- Fluxos de trabalho referenciados na saída segura [`dispatch-workflow`](/gh-aw/reference/safe-outputs/#workflow-dispatch-dispatch-workflow).
- Arquivos declarados no campo [`resources:`](/gh-aw/reference/frontmatter/#resources-resources) do frontmatter do fluxo de trabalho (fluxos de trabalho complementares, ações personalizadas).

> [!NOTE]
> Verifique cuidadosamente se o fluxo de trabalho vem de uma fonte confiável e se é apropriado para seu uso no seu repositório. Revise o conteúdo do fluxo de trabalho e entenda o que ele faz antes de adicioná-lo ao seu repositório.

> [!NOTE]
> Fluxos de trabalho marcados com `private: true` em seu frontmatter não podem ser adicionados a outros repositórios. Tentar fazer isso resultará em erro. Veja [Fluxos de Trabalho Privados](/gh-aw/reference/frontmatter/#private-workflows-private) para detalhes.

## Usando um Agente para Importar e Adaptar um Fluxo de Trabalho

Você pode usar um agente de codificação para importar um fluxo de trabalho de outro repositório e adaptá-lo para o seu. O agente lê o fluxo de trabalho de origem, personaliza a configuração específica do repositório (labels, atribuídos, nomes de branch, permissões) e configura o repositório — incluindo a inicialização, se necessário.

> [!TIP]
> Use esta abordagem quando quiser personalizar significativamente um fluxo de trabalho antes de usá-lo. Para importações diretas sem modificação, use [`gh aw add`](#adicionando-fluxos-de-trabalho) ou [`gh aw add-wizard`](#adicionando-fluxos-de-trabalho) em vez disso.

### Agente de Codificação

Siga estas etapas para importar e adaptar um fluxo de trabalho usando VSCode, Claude, Codex ou Copilot no seu terminal.

1. **Inicie seu agente de codificação** no contexto do seu repositório.

2. **Insira o seguinte prompt**, substituindo `SOURCE_WORKFLOW`, `OWNER` e `REPO` pelo fluxo de trabalho que você deseja importar:

   ```text wrap
   Inicialize este repositório para GitHub Agentic Workflows usando https://raw.githubusercontent.com/github/gh-aw/main/install.md

   Em seguida, importe e adapte o fluxo de trabalho SOURCE_WORKFLOW de OWNER/REPO. A fonte está em https://github.com/OWNER/REPO/blob/main/workflows/SOURCE_WORKFLOW.md.

   Adapte o fluxo de trabalho para este repositório: atualize quaisquer labels, atribuídos, nomes de branch e permissões para corresponder à estrutura deste projeto. Mantenha o propósito geral e a lógica do fluxo de trabalho intactos.
   ```

   Você pode adicionar o quanto de contexto extra, restrições ou objetivos de personalização após a última linha conforme precisar.

3. **Configure segredos necessários** se você ainda não o fez. Veja [Motores](/gh-aw/reference/engines/) para os segredos que o motor escolhido requer.

Após o agente terminar, revise o fluxo de trabalho adaptado, mescle o pull request e dispare uma execução a partir da aba Actions ou com `gh aw run`.

## Atualizando Fluxos de Trabalho

Ao adicionar um fluxo de trabalho, uma entrada `source:` de rastreamento lembra de onde ele veio. Você pode manter os fluxos de trabalho sincronizados com seus repositórios de origem:

```bash wrap
gh aw update                           # atualizar todos os fluxos de trabalho
gh aw update ci-doctor                 # atualizar fluxo de trabalho específico
gh aw update ci-doctor issue-triage    # atualizar vários
```

Use as flags `--major`, `--force`, `--no-merge`, `--engine` ou `--verbose` para controlar o comportamento da atualização. Versões semânticas (ex: `v1.2.3`) atualizam para a última versão compatível dentro da mesma versão principal. Referências de branch atualizam para o último commit. Referências de SHA atualizam para o último commit na branch padrão. As atualizações usam merge de 3 vias por padrão para preservar alterações locais; use `--no-merge` para substituir pela versão upstream. Quando ocorrerem conflitos de merge, resolva manualmente os marcadores de conflito e execute `gh aw compile`.

## Importações

Importe componentes reutilizáveis usando o campo `imports:` no frontmatter. Os caminhos de arquivo são relativos à localização do fluxo de trabalho:

```yaml wrap
---
on: issues
engine: copilot
imports:
  - shared/common-tools.md
  - shared/security-setup.md
  - shared/mcp/tavily.md
---
```

Durante o `gh aw add`, as importações são expandidas para rastrear o repositório de origem (ex: `shared/common-tools.md` torna-se `githubnext/agentics/shared/common-tools.md@abc123def`).

Importações remotas são automaticamente armazenadas em cache em `.github/aw/imports/` por SHA de commit. Isso permite a compilação de fluxo de trabalho offline após o download das importações. O cache é compartilhado entre diferentes refs que apontam para o mesmo commit, reduzindo downloads redundantes.

Veja [Referência de Importações](/gh-aw/reference/imports/) para formatos de caminho, semântica de merge e comportamento específico de campo.

## Importando Arquivos de Agente

Arquivos de agente fornecem instruções e comportamentos de IA especializados. Veja [Importando Arquivos de Agente Copilot](/gh-aw/reference/copilot-custom-agents/) para detalhes sobre como criar e importar arquivos de agente de repositórios externos.

## Exemplo: Fluxo de Trabalho Modular com Importações

Crie uma configuração de servidor Model Context Protocol (MCP) compartilhada em `.github/workflows/shared/mcp/tavily.md`:

```yaml wrap
---
mcp-servers:
  tavily:
    url: "https://mcp.tavily.com/mcp/?tavilyApiKey=${{ secrets.TAVILY_API_KEY }}"
    allowed: ["*"]
network:
  allowed:
    - mcp.tavily.com
---
```

Referencie-o no seu fluxo de trabalho para incluir o servidor MCP Tavily junto com outras ferramentas:

```yaml wrap
---
on:
  issues:
    types: [opened]
imports:
  - shared/mcp/tavily.md
tools:
  github:
    toolsets: [issues]
permissions:
  contents: read
---

# Agente de Pesquisa
Realize pesquisa na web usando Tavily e responda a issues.
```

**Resultado**: O fluxo de trabalho compilado inclui tanto o servidor MCP Tavily da importação quanto as ferramentas do GitHub do fluxo de trabalho principal, com permissões de rede mescladas automaticamente para permitir acesso tanto a `mcp.tavily.com` quanto a endpoints da API do GitHub.

## Melhores Práticas

Use versionamento semântico para fluxos de trabalho estáveis e arquivos de agente, branches para desenvolvimento e SHAs de commit para imutabilidade.
